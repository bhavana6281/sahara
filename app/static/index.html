How Sahara Works

Key features

  • Natural language queries — “ICU facilities with high trust in Bihar” instead of hunting through dropdowns
  • AI translates, never AI-judges — Llama 4 Maverick converts your question into a validated filter object; it never writes SQL and never touches the trust score
  • Interactive map — Leaflet map color-coded by trust level (Facility Trust Desk) or coverage status (Medical Desert Planner)
  • Deterministic trust scoring — every capability claim (ICU, surgery, oxygen…) is checked against contradiction rules in Spark, precomputed and auditable, not judged live by an LLM
  • Data desert vs. medical desert — the map never confuses “we don't know” with “confirmed absent care”
  • Leverage-scored review queue — ranks records by how much fixing them would improve the whole dataset, not just by raw trust score
  • Persisted decisions — planner notes/overrides and reviewer decisions survive restarts and redeployments
Experience Layer Databricks App — FastAPI backend, static vanilla-JS frontend, Leaflet maps. No build step.

Process flow — toggle a track

  1. Planner picks a capability + region, or types a plain-English question
  2. NL path: LLM (Llama 4 Maverick) emits a small validated JSON filter — capability/state/district/trust level — never SQL; unrecognized values are dropped to null, never guessed
  3. NL path, LLM unavailable: zero-LLM keyword fallback over the same allowlists — NL search still returns results, never a hard failure
  4. SQL Warehouse runs a parameterized query over facility_trust (capability match + state/district, including spelling-variant and pincode-recovered matches)
  5. Results ranked by trust_score, split into High/Medium/Low confidence tabs
  6. NL path: LLM writes a 2–3 sentence summary over the already-finalized results — cosmetic only, cannot change what was returned
  7. Planner expands a facility → cited contradictions + full capability evidence (all precomputed, just looked up)
  8. Planner note/override → written to planner_actions, persists across sessions and deployments
  1. Planner picks a capability + optional state
  2. One SQL statement: a WITH CTE computes each district's centroid (mean lat/lng of its facility_trust rows), state-scoped when a state filter is set so it stays fast
  3. LEFT JOIN to district_desert for the chosen capability — districts with no geolocatable facilities still return (not silently dropped)
  4. Map renders one marker per district, colored covered / medical desert / data desert — a deliberately different hue family from trust-level colors
  5. Popup shows facility count behind the aggregate, coverage ratio, and an explicit “approximate centroid, not a boundary” caveat
  6. Honest “N of M districts plotted” count — never silently hides what couldn't be mapped
  1. Summary query computes 4 headline numbers over facility_trust (total, with contradictions, low-trust, with missing evidence)
  2. Leverage query scores every facility: contradictions × 3 + missing supports × 2 + claims a critical capability (ICU/surgery/oncology/trauma/neonatal) × 5 + trust-level weight
  3. Top 200 by leverage_score, worst first — not naive low-trust sorting (98.9% of facilities have ≥1 contradiction, so that alone doesn't discriminate)
  4. Reviewer expands a record → the same cited-evidence view as the Trust Desk
  5. Reviewer records a decision: confirmed_issue / looks_fine / needs_field_check / corrected, plus an optional note
  6. Decision persists to review_decisions; queue can filter to unreviewed-only on reload — AI surfaces, a human decides, nothing auto-corrects
